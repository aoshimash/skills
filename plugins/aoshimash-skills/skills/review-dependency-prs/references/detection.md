# Composite Bot-PR Detection

Detailed procedure for Phase 1's detection step. The goal is to identify dependency-bot PRs reliably even when bots run under non-obvious accounts or multiple bot instances coexist in one repo.

## Why a single signal is not enough

A naive "author is `renovate[bot]`" check misses real setups:

- **Renovate runs in multiple modes.** Renovate Cloud (the Mend-hosted app) authors PRs as `renovate[bot]`, but a **self-hosted** Renovate runs under whatever account its token belongs to — often a generic CI/automation user with no "renovate" in the name.
- **Multiple instances coexist.** A repo can have Renovate Cloud *and* a self-hosted Renovate (e.g. for private registries), each producing PRs. Dependabot may also be enabled for security updates only. The same dependency can appear from two sources.
- **Accounts get renamed / proxied.** Org policies, GitHub Apps, and bot proxies change the visible author.

So detect with an **OR across multiple signals**, then **refine against the repo's own bot config**.

## Signals (OR across these)

A PR is a candidate dependency-bot PR if **any** of these match:

| Signal | What to check | Examples |
|---|---|---|
| **Author login pattern** | The PR author login / type. | `renovate[bot]`, `app/renovate`, `dependabot[bot]`, `app/dependabot`, plus self-hosted automation accounts identified from repo config. |
| **Branch prefix** | The head branch name. | `renovate/...`, `dependabot/...`, or the custom `branchPrefix` configured in `renovate.json5`. |
| **PR-body signature** | Marker text the bot embeds in the body. | Renovate: the Mend footer ("This PR was generated by Mend Renovate") or the `<!--renovate-debug-->` HTML comment block. Dependabot: the "Dependabot commands" / compatibility-score markers. |
| **Labels** | Labels the bot applies. | `dependencies`, `renovate`, `dependabot`, or the custom labels configured in the bot config. |

See [platform-github.md](platform-github.md) for the `gh` commands to read author, branch, body, and labels.

## Refine with repo bot config

Phase 0 read the bot config; use it here to make detection precise:

- `renovate.json5` / `.github/renovate.json` → the actual `branchPrefix`, configured `labels`, and (for self-hosted) clues to the bot identity. If `branchPrefix` is customized (e.g. `deps/`), the default `renovate/` check would miss every PR — the config is authoritative.
- `.github/dependabot.yml` → which ecosystems Dependabot covers and what labels/prefixes it uses.

If the config contradicts a default assumption, trust the config.

## Multiple instances and dedupe

Assume more than one bot may be active:

1. Detect all candidates across all signals.
2. **Dedupe by the underlying change**, not by author. If two PRs bump the same dependency to the same version (e.g. one from Renovate Cloud, one from self-hosted), they are duplicates — flag the pair, keep one, and note the other should be closed. This overlaps with the conflict detection in workflow.md step 1-2.
3. Keep distinct updates separate even if from the same bot.

## Ambiguity → confirm with the user

Detection is a filter, not a verdict. When a candidate is borderline — an unfamiliar automation account, a PR that matches a branch prefix but looks hand-authored, a body with no clear bot signature — **do not silently include or exclude it.** Present the ambiguous candidate(s) to the user with the signals that matched, and confirm whether to treat them as dependency-bot PRs.

This matters because the whole skill operates on the assumption that these are *machine-generated dependency updates*. Pulling a human's feature PR into the dependency loop, or skipping a real update from an unrecognized self-hosted bot, both cause real harm.

## Output of detection

A confirmed, deduped set of dependency-bot PRs, each annotated with:

- Source bot (Renovate Cloud / self-hosted Renovate / Dependabot / other).
- The signals that matched (useful when explaining the candidate set to the user).
- Any duplicate/superseded relationship to another PR in the set.

This set feeds conflict/ordering detection and the processing-order proposal in workflow.md.
