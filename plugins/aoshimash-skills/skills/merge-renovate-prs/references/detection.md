# Composite Renovate-PR Detection

Detailed procedure for Phase 1's detection step. The goal is to identify Renovate PRs reliably even when Renovate runs under a non-obvious account or multiple Renovate instances coexist in one repo. **Renovate only:** PRs from other dependency bots (e.g. Dependabot) are out of scope — do not process them; note their presence in the report.

## Why a single signal is not enough

A naive "author is `renovate[bot]`" check misses real setups:

- **Renovate runs in multiple modes.** Renovate Cloud (the Mend-hosted app) authors PRs as `renovate[bot]`, but a **self-hosted** Renovate runs under whatever account its token belongs to — often a generic CI/automation user with no "renovate" in the name.
- **Multiple instances coexist.** A repo can have Renovate Cloud *and* a self-hosted Renovate (e.g. for private registries), each producing PRs. The same dependency can appear from two sources.
- **Accounts get renamed / proxied.** Org policies, GitHub Apps, and bot proxies change the visible author.

So detect with an **OR across multiple signals**, then **refine against the repo's own Renovate config**.

## Signals (OR across these)

A PR is a candidate Renovate PR if **any** of these match:

| Signal | What to check | Examples |
|---|---|---|
| **Author login pattern** | The PR author login / type. | `renovate[bot]`, `app/renovate`, plus self-hosted automation accounts identified from repo config. |
| **Branch prefix** | The head branch name. | `renovate/...`, or the custom `branchPrefix` configured in `renovate.json5`. |
| **PR-body signature** | Marker text Renovate embeds in the body. | The Mend footer ("This PR was generated by Mend Renovate") or the `<!--renovate-debug-->` HTML comment block. |
| **Labels** | Labels the bot applies. | `dependencies`, `renovate`, or the custom labels configured in the Renovate config. |

See [platform-github.md](platform-github.md) for the `gh` commands to read author, branch, body, and labels.

## Refine with the Renovate config

Phase 0 read the Renovate config; use it here to make detection precise:

- `renovate.json5` / `.github/renovate.json` → the actual `branchPrefix`, configured `labels`, and (for self-hosted) clues to the bot identity. If `branchPrefix` is customized (e.g. `deps/`), the default `renovate/` check would miss every PR — the config is authoritative.

If the config contradicts a default assumption, trust the config.

## Multiple instances and dedupe

Assume more than one Renovate instance may be active:

1. Detect all candidates across all signals.
2. **Dedupe by the underlying change**, not by author. If two PRs bump the same dependency to the same version (e.g. one from Renovate Cloud, one from self-hosted), they are duplicates — flag the pair, keep one, and note the other should be closed. This overlaps with the conflict detection in workflow.md step 1-2.
3. Keep distinct updates separate even if from the same instance.

## Ambiguity → exclude (autonomous) or confirm (interactive)

Detection is a filter, not a verdict. When a candidate is borderline — an unfamiliar automation account, a PR that matches a branch prefix but looks hand-authored, a body with no clear Renovate signature — **do not silently include it.**

- **Autonomous:** exclude the ambiguous candidate from the run and list it in the final report with the signals that matched. Never auto-merge a PR you are not sure is machine-generated.
- **Interactive:** present the ambiguous candidate(s) to the user with the signals that matched, and confirm whether to treat them as Renovate PRs.

This matters because the whole skill operates on the assumption that these are *machine-generated dependency updates*. Pulling a human's feature PR into the dependency loop, or skipping a real update from an unrecognized self-hosted instance, both cause real harm.

## Output of detection

A confirmed, deduped set of Renovate PRs, each annotated with:

- Source instance (Renovate Cloud / self-hosted Renovate).
- The signals that matched (useful when explaining the candidate set).
- Any duplicate/superseded relationship to another PR in the set.

Plus a list of **out-of-scope bot PRs** noticed along the way (e.g. Dependabot) for the report.

This set feeds conflict/ordering detection and the processing-order proposal in workflow.md.
